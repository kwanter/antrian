# Plan: Fix `tv.html` UI/UX for Samsung TV display

## Goal

Make the legacy Samsung TV display (`frontend/public/tv.html`) readable, stable, and visually clean on Samsung UA55MU6100 browser. Main issue: loket/counter display per layanan is messy/cluttered.

## Context

- Normal Next.js `/display?tv=1` gets stuck on old Samsung Tizen browser at `Memuat...`.
- Legacy static fallback was added: `frontend/public/tv.html`.
- It uses plain HTML/CSS/JS + XHR polling for old TV compatibility.
- Current layout:
  - 62% video left, 38% layanan grid right.
  - Layanan cards use 2-column grid.
  - Each card shows current ticket + counter + recent rows.
- Problem likely caused by:
  - too many layanan squeezed into fixed 2-column side panel,
  - loket/counter labels have inconsistent length,
  - recent rows consume vertical space,
  - grid card heights shrink unpredictably,
  - old TV CSS engine handles `grid`/flex overflow less well.

## Proposed UX direction

Prioritize clarity over density.

1. Keep video left.
2. Right panel should show only the most important data:
   - layanan name/code
   - current ticket number
   - assigned/current loket as a badge
3. Move recent/history to a compact bottom strip or hide when too many layanan.
4. Make all cards fixed/min predictable height.
5. Use simple CSS: flex/block/table-like layout; avoid fragile nested grid/overflow combinations for Tizen 3.
6. Add adaptive modes based on number of layanan:
   - 1-2 layanan: big cards, full details.
   - 3-4 layanan: 2x2 cards, compact recent.
   - 5+ layanan: single-column/compact list, no recent rows, bigger current ticket readability.

## Implementation steps

### 1. Inspect prod data shape

Read real API data before editing layout:

```bash
curl -s http://192.168.5.14/api/v1/layanans | jq '.data[] | {id,name,code,counter_id,counter}'
curl -s 'http://192.168.5.14/api/v1/layanans/1/queues?status=called,serving,completed&date=YYYY-MM-DD' | jq '.data[0:3]'
```

Confirm:
- how many layanan are active,
- whether `layanan.counter` exists,
- whether queue counter name exists,
- common text lengths.

### 2. Refactor right panel layout in `frontend/public/tv.html`

Likely change only this file.

CSS changes:
- Replace `.grid` with `.queue-list` style that supports stable card rendering.
- Use explicit card variants via body/class:
  - `.mode-few`
  - `.mode-normal`
  - `.mode-many`
- Ensure card internals use simple blocks:
  - `.service-row`
  - `.service-title`
  - `.service-code`
  - `.ticket-row`
  - `.ticket`
  - `.loket-badge`
- Use ellipsis for long layanan/loket names.
- Reduce/remove `.recent` in many-mode.

Suggested visual hierarchy:

```text
[LAYANAN NAME] [CODE]
      A001
   LOKET 1
```

For many-mode:

```text
LAYANAN NAME     A001     LOKET 1
```

### 3. Fix JS rendering logic

In `renderLayanans(list)`:
- set layout class based on `list.length`:
  - `document.body.className = list.length > 4 ? 'mode-many' : list.length <= 2 ? 'mode-few' : 'mode-normal'`
- build card with clearer HTML structure.
- avoid large recent rows if `list.length > 4`.

In `updateQueue(l, card)`:
- determine loket name priority:
  1. `cur.counter.name`
  2. `l.counter.name`
  3. `display.settings.counter_id ? 'Loket terpilih' : 'Semua loket'`
- always update dedicated elements, not raw indexes only:
  - `.js-ticket`
  - `.js-loket`
  - `.js-status`
  - `.js-recent`
- if no current queue:
  - ticket `-`
  - loket badge remains assigned/default loket, not “Menunggu...” mixed with counter.
  - status text `Menunggu antrian`.

## Risks / tradeoffs

- Old Samsung browser may have weak CSS Grid support; prefer simple flex/block.
- Too many layanan cannot all be shown with big tickets; adaptive compact mode required.
- Polling each layanan every 3s can be heavy if many layanan. If needed, later add a single aggregated display endpoint.
- Announcer/TTS remains limited on UA55MU6100; separate issue from layout.

## Acceptance criteria

- `tv.html` loads on Samsung TV.
- Layanan/loket area is visually clean.
- Each layanan shows exactly one clear current loket badge.
- No text overlap/overflow in 1366x768 and 1920x1080.
- Display updates within 3s after queue call.
- Prod `http://192.168.5.14/tv.html` returns 200.
